1. Introduo

O escopo do projeto dividido em trs partes. Primeiramente, so capturados dados via scripts na linguagem python do IBGE e das plataformas Spotify e Spotify Charts. Aps obteno, tratamento e cruzamento dos dados, ento construdo, em linguagem MySQL, um banco de dados seguindo uma modelagem dimensional Entidade Relacionamento. Por fim, criado uma aplicao Web integrada ao banco utilizando as linguagens Python, MySQL, HTML, CSS e JavaScript. Todo o trabalho desenvolvido pode ser encontrado neste repositrio.

2. Dados

Com o objetivo de construir um banco de dados com msicas presentes em rankings nacionais do Spotify, a primeira fonte foi o Spotify Charts, um site feito por fs que apresenta ranking de msicas mais tocadas no Spotify por regio ou por gnero, no qual conseguimos dados dos rankings divididos por cidades do Brasil. Com isso, capturou-se microdados das msicas presentes nesses rankings utilizando como fonte API's fornecidas pela Plataforma de Desenvolvimento do Spotify. Por fim, foram capturados, atravs de uma API disponibilizada pelo IBGE, dados sobre a regio de cada cidade que contm ranking para complementar as informaes uti.

2.1 Obteno

Foi utilizado um Jupyter Notebook com linguagem Python com a biblioteca Selenium, realizando web scraping, para capturar dados do site Spotify Charts referentes a rankings por semana de cada cidade brasileira presente no site, incluindo os identificadores do Spotify de cada msica.

Com esses identificadores, a captao dos microdados das msicas na plataforma Spotify se d tambm por Jupyter Notebook com linguagem Python que se comunica com a plataforma via requests s APIs disponibilizadas, passando como parmetro inicial os identificadores. Dessa fonte, conseguimos diversos atributos das msicas, lbuns e artistas e, desses, foram selecionados os mais relevantes ao projeto para integrar a modelagem.

Os microdados do IBGE tambm foram obtidos da mesma forma que os do Spotify, via requests API disponibilizada pelo instituto, capturando os microdados apenas das cidades obtidas do Spotify Charts.

2.2 Cruzamento

Com todos os dados capturados e salvos em arquivos .CSV, as tabelas so lidas por um Jupyter Notebook com linguagem Python, utilizando a biblioteca pandas, e o cruzamento feito. 3. Modelagem

A partir do momento que os dados foram obtidos, entramos na fase da modelagem. Para isso, foi definido as entidades e depois as relaes entre elas.

3.1 Modelagem Conceitual

Considerando que a ideia envolvia a recomendao de msicas, a primeira e principal entidade modelada foi a de Msica. A ela, ento, foi atriuido um ID que a identifica, depois o seu Nome, Durao e Popularidade e, ento, as propriedades da msica - Explcito, Volume, Danabilidade, Energia e Valncia - que so utilizados no clculo de recomendao das msicas.

Com a Msica criada, chegamos s tabelas de lbum, com um identificador, nome, data e a imagem da capa; o Artista, com identificador, nome, nmero de seguidores, popularidade e gneros musicais e o Ranking, com identificador e a localidade, dividida em Cidade, Estado e Regio.

No caso dos gneros musicais do Artista, primeiro foi pensado modelar como um atributo multivalorado, para que a modelagem apresentasse um atributito desse tipo. Contudo, foi possvel adquirir mais de uma imagem por artista e assim o atributo multivalorado Imagem para artista foi criado, o que possibilitou ao gnero ser uma nova entidade com cardinalidade (1,n), j que um gnero s est na nossa base se possui pelo menos um artista o apresenta. J pensando na presena de um atributo composto, foi utlizado a localidade dos Rankings.

Depois que todas as entidades foram concludas, foram formadas as relaes entre elas. Assim, foi definida a relao Contm entre Ranking e Msica, com cardinalidade (1,n) em ambos os lados, pois cada ranking contm pelo menos uma msica e nosso banco s contm dados de msicas que esto presentes em pelo menos um ranking, e que possui os atributos Posio (posio da msica no ranking) e Data (indicando em qual ranking, separado em semanas, encontra-se uma dada msica).

Alm disso, temos a relao Tem entre lbum e Msica, onde cada msica est em um nico lbum, enquanto lbuns podem possuir vrias msicas; a relao possui o atributo Faixa, que indica qual a faixa que a msica ocupa no lbum. Por fim, temos o elacionamento Performa entre lbum e Artista, que tambm possui cardinalidade (1,n) em ambos os lados, j que o lbum pode ser de um mais artistas e nosso banco s contm artista que est relacionado a pelo menos um lbum, podendo ter vrios. Abaixo, se encontra a Modelagem Conceitual desenvolvida no programa brModelo.

3.2 Modelagem Relacional

Com a modelagem conceitual concluda, foi usada a funo de converso do brModelo para criar uma verso inicial e, a partir da, foram tulizadas as regras de converso para fazer quaisquer modificaes necessrias. Neste passo, ento, as relaes Performa e Contm, alm do atributo Gnero de Artista foram transformados em tabelas separadas, utilizando chaves estrangeiras para se relacionar com as entidades de origem. Enquanto isso, a relao Tem foi substituda pela adio, na tabela Msica, de uma chave estrangeira para o ID de lbum, alm do atributo Faixa.

5.1 Funcionalidades

A aplicao apresenta duas funcionalidades principais: a primeira a recomendao de msicas, onde o usurio pode entrar com o nome completo ou parcial de um cantor, lbum, msica ou cidade, sendo necessrio escolher por qual parmetro ser realizada a recomendao, caso o termo buscado esteja presente na base na categoria selecionada - no caso positivo, so retornadas vinte msicas que o algoritmo ingnuo considerou mais semelhantes, comparando as propriedades volume, danabilidade, energia e valncia.

Na funcionalidade de recomendao, caso a entrada seja uma msica, a aplicao busca as propriedades da faixa dada e os passa para o algoritmo que calcula a distncia entre a msica e todas as outras presentes na base, retornando as vinte com menor distncia. Caso seja artista, lbum ou cidade, a aplicao busca por todas as faixas que apresentam o valor dado no tipo especificado, calcula a mdia de cada propriedade, para ento passar essas mdias como entrada para algoritmo. Ento, se o usurio der um nome de um lbum por exemplo, primeiro sero resgatadas todas as msicas que pertencem ao lbum, depois feito uma mdia das msicas para cada propriedade e essas mdias que so passadas para o algoritmo que retornar as faixas com menores distncias. A busca por artista tem um diferencial em que pode-se entrar com um ou dois artistas separados por ;. Nesse caso, primeiro junta-se todas as msicas pertencentes aos artistas dados para ento calcular-se a mdia.

H ainda a possibilidade de uma busca personalizada para recomendao, onde o usurio pode especificar atravs de uma entrada por slider qual o nvel de volume, danabilidade, energia e valncia que ele deseja nas msicas, onde a aplicao pega essas entradas, converte para parmetros equivalentes s medidas das propriedades da msica e os passa para o algoritmo. Sendo possvel ainda nessa opo selecionar o critrio das msicas no serem explcitas.

J a segunda funcionalidade se resume em apresentar o ranking semanal das msicas mais populares por cidades brasileiras, estado ou regio, sendo que o spotify, atualmente, s fornece esse ranking para as cidades: Belo Horizonte, Belm, Braslia, Campinas, Campo Grande, Cuiab, Curitiba, Florianpolis, Fortaleza, Goinia, Manaus, Porto Alegre, Recife, Rio de Janeiro, Salvador, So Paulo e Uberlndia. H ainda a opo de visualizar o ranking Local Pulse.

Para os rankings das cidades, a aplicao apenas busca as msicas e suas respectivas posies para o tipo de ranking escolhido na cidade e semana especificadas, retornando suas informaes incluindo a imagem da capa do lbum recuperada do banco de dados e um player fornecido pelo Spotify para se tocar um pedao da msica ou ela completa caso o usurio esteja logado no servio no mesmo navegador.

J para estado e regio, como o Spotify Charts no fornece o nmero de reprodues de cada msicas, apenas suas posies, a aplicao busca os rankings das cidades pertencentes localidade e faz um rankeamento personalizado, classificando de acordo com a quantidade de cidades em que a msica entrou no ranking e as posies mais altas ocupadas nesses rankings para desempate.

Como a base de dados bem pequena com relao ao nmero de msicas e o algoritmo para recomendao elaborado bastante ingnuo, considerando ainda poucos parmetros e condies, as recomendaes no so assertivas no momento, mas conseguimos elaborar as funcionalidades que idealizamos e a aplicao pode vir a ser usada para futuras implementaes e melhorias.

5.2 Consultas

Para obtermos resultados das duas funcionalidades da nossa aplicao, recomendao e ranqueamentos, foram utilizadas diversas consultas ao banco de dados formado. Abaixo, so exibidas algumas:

  • Consulta envolvendo apenas as operaes de seleo e projeo:

Essa consulta seleciona as informaes de - Danceabilidade, Energia, Valencia e Volume, presentes na tabela musica para determinada msica informada pelo usurio na aplicao.

SELECT Danceabilidade, Energia, Valencia, Volume
FROM musica WHERE LOWER(musica.Nome) LIKE LOWER("{}");
  • Consulta envolvendo a juno de apenas duas relaes:

Essa consulta seleciona e obtm a mdia dos campos - Danceabilidade, Energia, Valencia e Volume, presentes na tabela musica para determinado lbum de entrada do usurio na aplicao. Como a informao de lbum no est presente na tabela de msica, foi necessrio realizar um join das duas tabelas, para obter as propriedades das msicas para o lbum selecionado pelo usurio.

SELECT AVG(Danceabilidade), AVG(Energia), AVG(Valencia), AVG(Volume)
FROM musica INNER JOIN album ON musica.ID_Album = album.ID
WHERE LOWER(album.Nome) LIKE LOWER("{}") GROUP BY album.ID;
  • Consulta envolvendo a juno externa de apenas duas relaes:

Essa consulta seleciona - musica.Nome, musica.ID, artista_albuns.Nome e Imagem_capa, presentes no join das tabelas de musica e artista_albuns, esta ltima foi criada anteriormente e foi composta de 2 joins entre outras 3 tabelas da base de dados.   • Consulta envolvendo a operao de unio:

A consulta de unio foi utilizada para funcionalidade de recomendao da aplicao, quando o usurio poder entrar com o nome de at dois artistas. Ento para isso, criamos duas subconsultas para obtermos as propriedades das msicas de cada um dos artistas. Sabemos que podamos utilizar na clusula WHERE a informao dos dois nomes de artista junto com o conectivo AND, mas optamos seguir dessa forma, para concluir a consulta exigida na definio do trabalho. Tendo as informaes de msicas de cada um dos artistas, unimos as tabelas e depois consultamos desta tabela unida, que nomeamos de unio, extraindo as informaes de mdia para cada uma das propriedades .

WITH
Cantor_1 AS (SELECT Danceabilidade, Energia, Valencia, Volume
FROM musica INNER JOIN album ON musica.ID_Album = album.ID
INNER JOIN artista_albums ON album.ID = artista_albums.ID_Album
INNER JOIN artista ON artista_albums.ID_Artista = artista.ID
WHERE LOWER(artista.Nome) LIKE LOWER("{singer1}") GROUP BY artista.ID ),
Cantor_2 AS (SELECT Danceabilidade, Energia, Valencia, Volume
FROM musica INNER JOIN album ON musica.ID_Album = album.ID
INNER JOIN artista_albums ON album.ID = artista_albums.ID_Album
INNER JOIN artista ON artista_albums.ID_Artista = artista.ID
WHERE LOWER(artista.Nome) LIKE LOWER("{singer2}") GROUP BY artista.ID ),
uniao AS (SELECT * FROM Cantor_1 UNION ALL SELECT * FROM Cantor_2)
SELECT AVG(Danceabilidade) AS avg_Danceabilidade, AVG(Energia) AS avg_Energia,
AVG(Valencia) AS avg_Valencia, AVG(Volume) AS avg_Volume
FROM uniao;

6. Execuo

necessrio ter o mysql instalado e executando, onde recomendado criar uma virtual-env em python para instalar o projeto para que no haja o risco de conflitos entre as bibliotecas. O projeto foi feito em python3 e caso no o tenha instalado, necessrio instal-lo antes de seguir para os prximos passos.

Com o python instalado, deve-se instalar as bibliotecas utilizadas especificadas no documento "requirements.txt", a instalao pode ser feita por:

$ pip3 install -r requirements.txt

Aps isso necessrio mudar as configuraes do arquivo config.yaml para seu nome de usurio e senha do mysql. O usurio geralmente "root".

OBS: Se voc estiver usando um hostou uma porta diferente do padro, apenas adicione junto com os outros campos um campo host: "meu_novo_host"e port: XXXXsubstituindo os valores pelos seus.

Com os parmetros certos preenchidos, primeiro deve-se executar o arquivo py que cria o banco de dados em seu servidor. Para isso execute o arquivo com "Database_creation.py" que pode ser executado rodando

$ python Database_Creation.py

na pasta do arquivo.

Com a base criada, basta apenas rodar o projeto com:

$ python app.py

A aplicao ficar ento disponvel no endereo endicado no cmd.
